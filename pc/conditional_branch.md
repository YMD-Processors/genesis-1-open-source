# **Genesis One CPU – Conditional Branch Handling (PC Module Update)**

---

## **1. Overview**

A **conditional branch** modifies the Program Counter (PC) **only when a runtime condition evaluates to true**.

Example instruction:

```
BEQ r1, r2, label
```

Semantics:

```
if (r1 == r2)
    PC = label
else
    PC = PC + 4
```

In Genesis One, this behavior is implemented using:

* External branch decision logic (Execute stage)
* PC arbitration logic (Fetch stage)

---

## **2. Architectural Principle**

> ⚠️ **The PC does NOT evaluate conditions.**

All decision-making occurs in the **Execute stage**, which outputs:

* `branch_taken`
* `branch_target`

The PC acts purely as a **control-flow router**.

---

## **3. Required Signals (Already Implemented)**

### **3.1 Branch Taken**

```verilog
input wire branch_taken;
```

| Value | Meaning          |
| ----- | ---------------- |
| 0     | Branch NOT taken |
| 1     | Branch taken     |

Generated by:

```
ALU / Branch Unit (Execute stage)
```

---

### **3.2 Branch Target**

```verilog
input wire [XLEN-1:0] branch_target;
```

* Precomputed address
* Typically:

```
branch_target = PC + immediate_offset
```

* Delivered already resolved to the PC

---

## **4. Genesis One PC Behavior (With Full Pipeline Context)**

Unlike a minimal CPU, Genesis One includes:

* Branch prediction
* Pipeline flush
* Interrupt/exception handling

So conditional branching exists **within a larger priority system**.

---

## **5. Updated Next-PC Arbitration (Accurate to Your RTL)**

### **Full Priority Order**

| Priority | Condition                      | Action               |
| -------- | ------------------------------ | -------------------- |
| 1        | Exception                      | `PC = vector_addr`   |
| 2        | Interrupt                      | `PC = vector_addr`   |
| 3        | Flush (mispredict recovery)    | `PC = flush_target`  |
| 4        | Stall                          | `PC = PC` (hold)     |
| 5        | Prediction valid               | `PC = predicted_pc`  |
| 6        | **Branch taken (conditional)** | `PC = branch_target` |
| 7        | Jump                           | `PC = jump_addr`     |
| 8        | Default                        | `PC = PC + 4`        |

---

## **6. Conditional Branch Rule (Within This System)**

The effective rule is:

```
if (branch_taken && !higher_priority_events)
    PC = branch_target
```

Expanded:

```
if exception
    PC = vector_addr
else if interrupt
    PC = vector_addr
else if flush
    PC = flush_target
else if stall
    PC = PC
else if prediction_valid
    PC = predicted_pc
else if branch_taken
    PC = branch_target
else if jump
    PC = jump_addr
else
    PC = PC + 4
```

---

## **7. Alignment Requirement**

All branch targets are aligned:

```verilog
aligned_branch = {branch_target[XLEN-1:2], 2'b00};
```

### Why:

* Ensures 4-byte instruction alignment
* Prevents illegal instruction fetch

---

## **8. Interaction with Branch Prediction**

Genesis One supports **speculative execution**:

### Case A: Prediction Used

```
prediction_valid = 1
→ PC follows predicted_pc
```

### Case B: Prediction Wrong

```
flush = 1
→ PC redirected to correct target
```

### Role of Conditional Branch:

* Actual `branch_taken` resolves correctness
* May trigger pipeline flush

---

## **9. Timing Example (Real Pipeline Behavior)**

Instruction stream:

```
0x0000  ADD
0x0004  BEQ r1,r2,0x20
0x0008  ADD   (speculative)
0x000C  ADD   (speculative)
```

---

### **Case 1: Branch Taken (Correct Prediction)**

| Cycle | prediction_valid | branch_taken | PC     |
| ----- | ---------------- | ------------ | ------ |
| 0     | 0                | 0            | 0x0000 |
| 1     | 1                | 0            | 0x0004 |
| 2     | 1                | 1            | 0x0020 |

---

### **Case 2: Misprediction**

| Cycle | Event                     | PC     |
| ----- | ------------------------- | ------ |
| 0     | Fetch BEQ                 | 0x0004 |
| 1     | Wrong prediction → 0x0008 |        |
| 2     | branch_taken = 1          |        |
| 3     | flush asserted            | 0x0020 |

## **10. Hardware Implementation**

Inside the PC module, conditional branching is part of a **multi-input MUX**:

```
                 +-----------------------------+
PC + 4 --------->|                             |
branch_target -->|                             |
jump_addr ------>|        NEXT PC MUX          |----> PC
predicted_pc --->|                             |
flush_target --->|                             |
vector_addr ---->|                             |
                 +-----------------------------+
                           ^
                           |
                    control logic
```

Where:

```
branch_taken selects branch_target
```

---

## **11. Pipeline Responsibility Split**

### **Execute Stage**

* Evaluates condition
* Computes branch target

### **PC (Fetch Stage)**

* Selects next PC
* Redirects instruction flow

---

## **12. Key Design Insights**

### ✅ Clean Separation of Concerns

* PC = control-flow router
* ALU = decision maker

---

### ✅ Zero Computation in PC

* No comparison logic
* No arithmetic for conditions

---

### ✅ Scalable Design

* Easily supports:

  * More branch types (BNE, BLT, etc.)
  * Advanced predictors
  * Out-of-order execution (future)

---

### ✅ Minimal Hardware Overhead

Conditional branch support adds only:

```
+ 1 MUX input (branch_target)
+ 1 control signal (branch_taken)
```
