# **Genesis One CPU – Alignment Enforcement Specification**

## **1. Definition**

In the **Genesis One CPU**, alignment enforcement guarantees that all instruction addresses generated by the Program Counter (PC) are **4-byte aligned**.

This means:

```text
PC[1:0] = 2'b00
```

Only valid instruction boundaries are allowed.

---

## **2. Valid vs Invalid Addresses**

### ✅ Valid PC Values (Aligned)

```text
0x0000
0x0004
0x0008
0x000C
```

### ❌ Invalid PC Values (Misaligned)

```text
0x0001
0x0002
0x0003
0x0005
```

The Genesis One PC **guarantees invalid values are never produced**.

---

## **3. Alignment Model in Genesis One**

Genesis One uses:

```text
Instruction width = 4 bytes
PC increment (COUNTER) = 4
```

Therefore:

```text
PC_next = PC + 4
```

This inherently preserves alignment during sequential execution.

---

## **4. Enforcement Strategy**

Genesis One implements **strict hardware alignment (masking)**.

All incoming addresses are aligned **before entering PC arbitration logic**.

### **Implementation (from your design)**

```verilog
aligned_addr = {addr[XLEN-1:2], 2'b00};
```

---

## **5. Alignment Coverage**

Alignment is enforced on **ALL external control-flow inputs**:

| Source     | Signal          | Alignment Applied |
| ---------- | --------------- | ----------------- |
| Reset      | `reset_vector`  | ✔                 |
| Branch     | `branch_target` | ✔                 |
| Jump       | `jump_addr`     | ✔                 |
| Flush      | `flush_target`  | ✔                 |
| Interrupt  | `vector_addr`   | ✔                 |
| Exception  | `vector_addr`   | ✔                 |
| Prediction | `predicted_pc`  | ✔                 |

### From your module:

```verilog
wire [XLEN-1:0] aligned_reset_vector = {reset_vector[XLEN-1:2], 2'b00};
wire [XLEN-1:0] aligned_predicted_pc = {predicted_pc[XLEN-1:2], 2'b00};
wire [XLEN-1:0] aligned_branch       = {branch_target[XLEN-1:2], 2'b00};
wire [XLEN-1:0] aligned_flush        = {flush_target[XLEN-1:2], 2'b00};
wire [XLEN-1:0] aligned_vector_addr  = {vector_addr[XLEN-1:2], 2'b00};
wire [XLEN-1:0] aligned_jump         = {jump_addr[XLEN-1:2], 2'b00};
```

---

## **6. Placement in Pipeline**

Alignment occurs **before next-PC selection**:

```text
Incoming Address
        ↓
Alignment Logic (mask lower 2 bits)
        ↓
Next PC Arbitration
        ↓
PC Register
```

This ensures:

* Arbitration logic only sees **valid addresses**
* No misaligned value can propagate

---

## **7. Sequential Execution Safety**

Sequential updates are inherently aligned:

```verilog
pc_plus = pc_reg + COUNTER;  // COUNTER = 4
```

Since:

```text
aligned PC + 4 → always aligned
```

No additional enforcement is required for:

* Normal instruction flow

---

## **8. No Alignment Exceptions (Design Choice)**

Unlike some architectures, Genesis One:

```text
DOES NOT raise misalignment exceptions
```

Instead:

```text
Misaligned inputs are silently corrected
```

### Example:

```text
Input branch_target = 0x1003
Aligned result      = 0x1000
```

---

## **9. Hardware Cost**

Alignment enforcement is extremely efficient:

```text
Cost = forcing 2 LSBs to zero
```

In hardware:

```text
addr[1] = 0
addr[0] = 0
```

No adders, comparators, or control logic required.

---

## **10. Design Rationale**

Genesis One chooses **masking over exception-based alignment** because:

### Advantages:

* Zero-cycle enforcement
* No pipeline stalls
* Simpler control logic
* Deterministic behavior
* Better suited for high-performance pipelines

### Trade-off:

* Silent correction may hide software bugs

---

## **11. Formal Alignment Rule**

For Genesis One:

```text
PC % 4 == 0
```

Or equivalently:

```text
PC[1:0] == 2'b00
```

---

## **12. Critical Property**

> **At no point in execution can the PC hold a misaligned address.**

This is guaranteed because:

1. All inputs are aligned
2. Sequential increments preserve alignment
3. PC register only loads aligned values
